  - **Phase 4 ✅ (ex-3, livrée 2026-05-09)** — rapprochement automatique facture ↔ transaction bancaire.
    - **Migration 0003** : audit trail (matched_at, matched_by 'auto'|'manual', matched_user_id FK profiles)
    - **Engine** `lib/engines/invoice-matching.ts` (pur) : pré-filtrage `kind='revenue'` + fenêtre `[due_at-7, due_at+30]` + tx-non-attribuée (Set in-memory) + amount HARD_BOUND ±20%. Scoring 4 dim : montant ±1% (0.40), fenêtre ±30j (0.20), client dans label (0.20), n° dans label (0.20). **Idempotence stricte** : skip toute invoice avec `matched_transaction_id != null`. **FIFO** sur invoices (`due_at ASC`, `id ASC`). **Tiebreakers candidats** : score DESC, |tx.date−due_at| ASC, tx.id ASC.
    - **3 branches explicites** :
      - score ≥ 0.90 ET |delta| ≤ 1% → auto-match (status='paid', paid_at=tx.date, matched_*, audit `matched_by='auto'`)
      - 0.60 ≤ score < 0.90 ET |delta| ≤ 1% → suggestion persistée (matched_transaction_id + match_confidence + audit, status reste pending)
      - tx < 0.99×invoice → underpayment (in-memory, **aucune écriture DB**, panneau « Paiements partiels suspectés »)
      - tx > 1.01×invoice → overpayment (idem, panneau « Paiements supérieurs attendus »)
    - **Auto-trigger silencieux** : appelé en interne par `createInvoice`, `bulkInsertInvoices`, `insertTransactions` après chaque succès → matching cross-sided (factures vs transactions, dans les deux sens).
    - **UI** : page dédiée `/invoices/suggestions` (3 sections distinctes + bouton « Relancer » + timestamp « Dernier rapprochement : il y a X minutes »). Sur `/invoices` : bouton « Annuler match » (Unlink) sur les paid invoices avec matched_transaction_id, qui appelle `unmatchPaid` (reset complet → pending, paid_at=null, matched_*=null, audit=null). Sidebar : entrée « Suggestions » sous « Factures ».
    - **Réversibilité** : `dismissMatch` (clear matched_*, audit ; status pending) pour les suggestions, `unmatchPaid` (reset complet) pour les paid.
    - **Limites V1 assumées** : (i) un dismissMatch peut être re-suggéré au prochain run (engine sans mémoire des refus) ; (ii) les anomalies sont in-memory only, perdues sur refresh ; (iii) auto-trigger silencieux = résultats visibles uniquement en visitant `/invoices/suggestions`.

- **1.8 — Clôture provisoire vs définitive (V1.5)** : la 1.7 actuelle traite la clôture comme un acte binaire et irréversible, ce qui ne colle pas à la pratique TPE/PME française. Le bilan officiel n'est validé qu'en mars/avril N+1 (réception des FAE/FNP, calcul des CCA/PCA, provisions, amortissements, remontées de l'expert-comptable), donc on doit pouvoir passer des écritures de régularisation/inventaire datées 31/12/N **après** une 1ère « clôture provisoire ». Conséquence post-E2E réel : Mag a clôturé 31/12/2025 puis a dû SQL-revert (escape var `app.allow_archive_modification`) sa propre clôture pour pouvoir continuer à imprimer/comptabiliser en avril/mai. La 1.7 telle qu'elle existe reste en place techniquement mais ne sera pas utilisée en pratique tant qu'on n'a pas la 1.8.
  - **Périmètre** : (a) nouveau champ `accounting_closures.status enum('provisional', 'final')` default `'provisional'` ; (b) flag `transactions.is_inventory_adjustment` + `invoices_emitted.is_inventory_adjustment` (type d'écriture « Régularisation/Inventaire » : CCA, PCA, FNP, FAE, provisions, amortissements) ; (c) trigger `prevent_modify_archived` adapté pour bypass si la closure correspondante est `status='provisional'` ET la row a `is_inventory_adjustment=true` (les écritures normales restent gelées même en provisoire) ; (d) bouton UI « Marquer comme définitive » sur une closure provisoire (action irréversible, role owner, recopie « DEFINITIVE » à saisir) ; (e) auto-bascule `provisional → final` après X mois (paramétrable par company, défaut 6 mois soit ~30 juin pour exercice civil) via cron Supabase ; (f) audit trail `inventory_adjustments_log(closure_id, row_id, row_table, user_id, amount_delta_cents, reason, created_at)` pour traçabilité expert-comptable.
  - **Note de planification** : ne pas attaquer avant que **1.6.5 + Quadrimarket + Simulations enrichies** soient livrés. C'est V1.5, pas V1. La 1.7 actuelle suffit pour les démos et les workflows simples ; la 1.8 est une dette UX/comptable connue mais pas bloquante pour les premiers users.
